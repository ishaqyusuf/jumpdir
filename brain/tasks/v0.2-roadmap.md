# v0.2 Roadmap: Onboarding, Runner Preference, and Jump Workflow

## Objective
Make the first-run experience self-guided and make daily usage faster:

- On first run, guide the user through setup.
- Let the user choose a preferred script runner without calling it a "default".
- Let users configure project directories interactively.
- Let users run scripts with either a saved runner preference or an explicit runner in the command.
- Add a path/jump workflow for `termcode <project>`.

## Naming Decision
Use **Preferred Runner** instead of "default script runner".

Reason: "default" sounds mandatory and global. "Preferred Runner" is softer and clearer: it is the runner `termcode` should try when the command does not explicitly name one.

Supported preferred runner values:

- `bun run`
- `pnpm run`
- `npm run`
- `yarn run`
- `none`

`none` means: do not assume a runner. If the command needs one, ask the user to configure a preferred runner or provide one explicitly.

Resolved implementation decisions:

- `auto` is not included in v0.2.
- First-run onboarding triggers only on plain `termcode`.
- `termcode <project>` prints the project path before shell integration.
- `termcode init zsh` is the shipped parent-shell jump integration.

## Target User Flows

### First Run
When the user runs `termcode` and no setup exists:

```sh
termcode
```

Show onboarding:

```text
Welcome to termcode.

Step 1: Choose your preferred runner
  1. bun run
  2. pnpm run
  3. npm run
  4. yarn run
  5. none

Step 2: Add project directories
  Example: ~/Documents/code ~/Desktop/projects
```

After directories are added, list discovered projects:

```text
Found 12 projects:
  gnd
  termcode
  my-app

What next?
  1. Add another directory
  2. Proceed
```

### Run Script With Preferred Runner
If the user configured `pnpm run`:

```sh
termcode gnd websites
```

Expected behavior:

```sh
cd <resolved-gnd-path>
pnpm run websites
```

### Run Script With Explicit Runner
If no preferred runner is configured, or the user wants to override it:

```sh
termcode gnd bun run websites
termcode gnd pnpm run websites
termcode gnd npm run websites
termcode gnd yarn run websites
```

Expected behavior:

```sh
cd <resolved-gnd-path>
<runner> run websites
```

### Missing Preferred Runner
If the user runs a script without a preferred runner:

```sh
termcode gnd websites
```

Expected output:

```text
termcode: no preferred runner is set.

Choose one:
  termcode runner set bun
  termcode runner set pnpm
  termcode runner set npm
  termcode runner set yarn

Or provide one for this command:
  termcode gnd bun run websites
```

### Jump To Project Path
Desired command:

```sh
termcode gnd
```

Important shell constraint: a child process cannot change the parent shell directory.

Recommended implementation:

1. `termcode gnd` prints the resolved project path by default.
2. Add shell integration so users can opt into real directory jumping:

```sh
eval "$(termcode init zsh)"
```

After shell integration:

```sh
termcode gnd
```

can run:

```sh
cd "$(command termcode path gnd)"
```

Alternative if we do not want shell integration yet:

```sh
cd "$(termcode path gnd)"
```

## Detailed Execution Plan

### Phase 1: Config Model
- Add `runner` config file under the existing config directory.
- Store the preferred runner as one token: `bun`, `pnpm`, `npm`, `yarn`, or `none`.
- Keep existing `roots` and `aliases` formats unchanged.
- Add helpers:
  - `get_runner_preference`
  - `set_runner_preference`
  - `has_completed_onboarding`
  - `mark_onboarding_complete`
- Validation:
  - `termcode runner get` prints the current preference.
  - `termcode runner set pnpm` writes `pnpm`.
  - invalid runner values fail clearly.

### Phase 2: Runner Commands
- Add command group:
  - `termcode runner get`
  - `termcode runner set <bun|pnpm|npm|yarn|none>`
  - `termcode runner clear` as an alias for `runner set none`
- Keep command names short and discoverable in `--help`.
- Validation:
  - tests for set/get/clear.
  - tests for invalid input.

### Phase 3: Interactive Onboarding
- Trigger onboarding only when:
  - config directory does not exist, or
  - no roots are configured and onboarding is not marked complete.
- Step 1: prompt for Preferred Runner.
- Step 2: prompt for project directories.
- After setting directories, list discovered projects.
- Prompt: `Add another directory` or `Proceed`.
- When complete, write an `onboarded` marker file.
- Validation:
  - test with piped input for first run.
  - test that onboarding does not repeat after completion.
  - test that `termcode --help` never triggers onboarding.

### Phase 4: Project Listing During Setup
- Reuse existing project scanner.
- After each directory update, show project count and names.
- Detect duplicate callable names during onboarding and explain aliases:

```text
Some project names are duplicated.
Use `termcode alias <name-or-path> <alias>` after setup to disambiguate.
```

- Validation:
  - test project listing after setup.
  - test duplicate messaging during setup.

### Phase 5: Script Command Parser
- Support both forms:
  - `termcode <project> <script> [args...]`
  - `termcode <project> <runner> run <script> [args...]`
- Recognize explicit runners only when argument 2 is one of `bun`, `pnpm`, `npm`, `yarn` and argument 3 is `run`.
- If explicit runner is present, use it.
- If explicit runner is not present:
  - use Preferred Runner when configured.
  - otherwise show the missing preferred runner guidance.
- Decide whether lockfile auto-detection remains as a fallback:
  - Done: `auto` is not offered in onboarding, and v0.2 does not guess from lockfiles.
- Validation:
  - `termcode gnd websites` uses preferred runner.
  - `termcode gnd bun run websites` overrides preference.
  - no preference produces the guidance message.
  - args pass through correctly after script name.

### Phase 6: Path and Jump Commands
- Add `termcode path <project>` to print only the resolved path.
- Decide base behavior for `termcode <project>`:
  - Recommended CLI behavior before shell integration: print project path.
  - Later shell integration can intercept this and `cd` there.
- Add `termcode init zsh` to print a shell function that wraps the binary and changes directory when only one project argument is provided.
- Optional later: `termcode init bash`.
- Validation:
  - `termcode path gnd` prints a single path and nothing else.
  - `termcode gnd` prints a path when no shell integration is installed.
  - generated zsh function syntax passes `zsh -n`.

### Phase 7: Documentation
- Update README:
  - onboarding screenshots/text.
  - Preferred Runner explanation.
  - explicit runner examples.
  - path/jump explanation and shell limitation.
- Update `brain/system/architecture.md`:
  - runner preference file.
  - onboarding marker.
  - command parser changes.
- Update `brain/features/termcode-cli.md`:
  - v0.2 behavior.
- Validation:
  - README examples match tests.

### Phase 8: Release Preparation
- Bump version from `0.1.0` to `0.2.0`.
- Update Homebrew template version URL.
- Run full test harness.
- Install locally from source and smoke test:
  - `termcode --help`
  - `termcode runner set pnpm`
  - `termcode path <project>`
  - `termcode <project> <script>`
